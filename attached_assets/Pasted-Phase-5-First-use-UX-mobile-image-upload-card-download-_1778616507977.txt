Phase 5 — First-use UX, mobile image upload, card download/share, and guided funnel fixes.

Do NOT add new gameplay features.
Do NOT modify penalty gameplay.
Do NOT modify Football IQ question content.
Do NOT modify progression economy.
Do NOT modify badge logic.
Do NOT change quiz scoring.
Do NOT redesign the whole app.

Context:
The app is now live on fangirlfc.live.
After real mobile testing, we found several launch-blocking UX issues:

1. Card image flow only allows taking a selfie. Users also need to upload an existing photo from gallery.
2. The first-use flow is not guided enough. Users need clearer steps from Quiz → Card → Football IQ → Penalty → Updated Card → Share.
3. Sign up is not visible enough, but we should not force sign up before the user experiences the product.
4. Card download does not work reliably on mobile.
5. Shared card link does not work when opened by another user/device.
6. The main card should feel like the central object that all flows update.

Goal:
Fix the first-use experience and make the card actually shareable on mobile.

## 4. Mobile card export/share fix

Audit the current export/download system.

Current issue:
Card download does not work reliably on mobile.

Implement robust mobile-friendly share/save behavior.

Required behavior:
Primary button should be:
“Share / Save card”

When clicked:
1. Generate the card image from cardRef.
2. If Web Share with files is supported:
   - share the PNG image file.
3. Else:
   - trigger download.
4. If download fails:
   - open the generated image in a new tab or modal.
   - show instruction: “Long press the image to save.”

Do not include outside page UI in the exported image.
Only cardRef should be exported.

Acceptance:
- Works on desktop.
- Works on mobile browser as much as possible.
- Provides fallback if direct download is blocked.
- User can still save the card manually.
- No console errors.

## 5. Public share link fix

Current issue:
Shared card link does not work properly when opened by another user/device.

Audit the current share link implementation.

We need a real public card share link.

Preferred architecture:
- When user clicks “Share card link”, create a shareId.
- Save the minimum card payload needed to render the card.
- Create route:
  /share/[shareId]
- Opening /share/[shareId] on another device should render the card publicly.

Important:
Do not depend on localStorage for public share links.
LocalStorage only works on the same device.

If Firebase/Firestore is already configured:
- create public_cards collection or reuse existing safe share storage if present.
- store only non-sensitive card data:
  - identity id
  - display name
  - selected template/vibe
  - team
  - card image/photo if already stored safely OR fallback to no photo if not possible
  - progress proof values
  - createdAt
- Do not store unnecessary personal data.
- Do not store private auth tokens.

If Firebase is not configured for this flow:
- report the safest minimal implementation path before coding.

Acceptance:
- User can create a share link.
- Opening the share link in another browser/device works.
- It renders the card or a share preview page.
- It does not rely on localStorage.
- If unauthenticated users cannot create public links safely, show sign-up prompt.
